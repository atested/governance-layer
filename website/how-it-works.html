How Atested works

From action request to auditable proof, in six steps. Every step produces a verifiable artifact. Nothing is taken on trust.

Action request

An AI agent requests an action through the Model Context Protocol. This could be writing a file, sending a message, reading data, or invoking any governed tool. The request includes the tool name, arguments, and the operator's stated intent.

Policy evaluation

The governance layer evaluates the action against your policy. Policies are defined in a capability registry that specifies which actions are allowed, under what constraints, and with what arguments. The evaluation produces a decision: ALLOW or DENY.

The evaluation is deterministic. The same action, with the same policy, always produces the same decision. This is what makes governance records replayable.

Decision record

Whether allowed or denied, the decision is recorded. The record captures the tool, the arguments, the policy decision, the timestamp, the operator identity, and a SHA-256 hash of the record contents.

decision record ALLOW 
{
  "tool":             "fs_write",
  "policy_decision":  "ALLOW",
  "timestamp_utc":    "2026-05-28T09:12:00Z",
  "user_identity":    "bearer:e1f2a3b4c5d67890",
  "record_hash":      "sha256:0a1b2c3d...",
  "prev_record_hash": "sha256:f0e1d2c3..."
}

Chain append

The record is appended to an immutable decision chain. Each record's hash includes the previous record's hash, creating a tamper-evident linked structure. If any record is modified or removed after the fact, the chain verification fails.

The chain is cryptographically signed with Ed25519 keys. An independent verifier can validate the entire chain without access to the governance system that produced it.

Attestation

At any point, the system can generate an attestation artifact that summarizes the governance activity: how many users, how many operations, which tool categories, the license status, and a hash binding the attestation to the chain.

attestation.json ATTESTATION 
{
  "attestation_version": "1.0",
  "unique_users":        5,
  "total_operations":    2104,
  "operations_by_category": {
    "FS_READ":  612,
    "FS_WRITE": 489,
    "FS_LIST":  301,
    "MSG_SEND": 187,
    "MSG_REPLY": 142
  },
  "license_tier":     "team",
  "organization_id":  "acme-engineering",
  "recommended_tier": "team",
  "artifact_hash":    "sha256:b4c2d5e6..."
}

Proof packet

A proof packet bundles everything into a single verifiable deliverable: the decision records, the policy that produced them, the attestation, and a manifest with cryptographic hashes of every component. An external verifier can validate the entire packet independently.

This is what you hand to an auditor, a customer, or a compliance review. It is not a report you wrote. It is evidence your system produced.

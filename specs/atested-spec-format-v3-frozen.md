# Atested Specification Format v3 (Frozen)

## Purpose

This document defines the standard format for all Atested specification documents. Specifications written in this format serve three functions simultaneously:

1. **Human reference** - readable documentation of what the system does
2. **Quality service input** - machine-parseable verification rules compiled by the spec compiler
3. **Test generation source** - build-time tests and regression gates derived from the same requirements

Every specification document that describes normative system behavior must conform to this format. Documents that are purely strategic, philosophical, or exploratory are exempt.

## Document Envelope

Every spec file begins with a YAML frontmatter block:

```yaml
---
spec_id: telemetry-v1
title: Telemetry System Specification
status: active
scope: governance-layer
last_verified: 2026-05-19
owner: greg
imports:
  - spec: chain-events-v1
    events: [EVT-TEL-001, EVT-TEL-002]
  - spec: tier-registry-v1
    schemas: [SCH-LIC-010]
---
```

Status values:

- `active` - normative, quality service verifies against this spec
- `suspended` - not under verification, not currently being built
- `draft` - under design, not yet normative
- `deprecated` - replaced by another spec, retained for reference

Imports declare versioned dependencies on other specs by element ID. The compiler resolves imports and rejects references to undefined elements. Coverage estimates are generated by the compiler from element statuses and do not appear in frontmatter.

## YAML Artifact Convention

Converted specifications and registries are stored as YAML artifacts, not Markdown documents with embedded YAML snippets. A YAML artifact uses top-level envelope fields followed by a single `elements` array. The Markdown frontmatter form shown above is retained for human-authored Markdown specs, but YAML artifacts replace frontmatter with ordinary top-level YAML keys.

Required top-level envelope fields:

```yaml
spec_id: telemetry-v1
title: Telemetry System Specification
status: active
scope: governance-layer
last_verified: 2026-05-19
owner: greg
imports:
  - spec: chain-events-v1
    events: [EVT-TEL-001, EVT-TEL-002]
elements:
  - element_type: requirement
    id: REQ-TEL-001
    ...
```

`imports` may be an empty array. `elements` is always present and contains every normative object in the artifact.

Every object in `elements` must include `element_type` as its discriminator. The compiler uses `element_type` to select the schema for the rest of the object. Do not use a top-level `type` field as the element discriminator; `type` remains available inside JSON Schema fragments and other nested structures where it has its normal schema meaning.

Allowed `element_type` values:

- `requirement`
- `data_schema`
- `endpoint_contract`
- `chain_event_schema`
- `state_machine`
- `negative_constraint`
- `workflow_contract`
- `schedule_contract`
- `webhook_contract`
- `derived_artifact`
- `configuration_entry`
- `privacy_boundary`

Registry artifacts use the same convention. For example, `chain-events-v1.yaml` is a YAML artifact whose `elements` array contains only `chain_event_schema` elements, while `tier-registry-v1.yaml` may contain data schemas and configuration entries.

## Component Identifiers

Every normative element declares which components produce or consume the behavior. Component identifiers are drawn from a fixed set:

| Identifier | Description |
|---|---|
| proxy | Governance proxy (mediates tool calls) |
| dashboard | Local dashboard server (Python/FastAPI) |
| browser-ui | Dashboard frontend rendered in operator's browser |
| governance-cli | Atested CLI tool |
| atested-com-worker | Cloudflare Worker at license.atested.com |
| version-worker | Cloudflare Worker at version.atested.com |
| stripe | Stripe external service |
| quality-service | The QA verification engine itself |

Components are declared as arrays:

```yaml
components:
  producers: [dashboard, proxy]
  consumers: [atested-com-worker]
```

## Stable Element IDs

Every normative element carries a stable ID. The QA chain, coverage manifest, cross-spec imports, and source traceability all reference elements by ID. IDs are unique across the entire spec corpus.

| Element type | ID prefix | Example |
|---|---|---|
| Requirement | REQ- | REQ-TEL-001 |
| Data schema | SCH- | SCH-TEL-001 |
| Endpoint contract | EP- | EP-TEL-001 |
| Chain event schema | EVT- | EVT-TEL-001 |
| State machine | SM- | SM-LIC-001 |
| Negative constraint | NEG- | NEG-SEC-001 |
| Workflow contract | WF- | WF-EXP-001 |
| Schedule contract | SCHED- | SCHED-UI-001 |
| Webhook contract | WH- | WH-WEB-001 |
| Derived artifact | DRV- | DRV-CHN-001 |
| Configuration entry | CFG- | CFG-TEL-001 |
| Privacy boundary | PRV- | PRV-TEL-001 |

Generated IDs follow a separate namespace. Privacy boundary auto-generated constraints use `NEG-AUTO-{PRV-ID}-{NNN}`. Domain codes are GOV, CHN, LIC, TEL, UI, SEC, EXP, COM, SIM, SIG, ADM, and WEB.

## Three-Status Model

Every normative element carries three independent status fields:

```yaml
status: active
implementation_status: implemented
verification_status: verified
```

`status` records normative intent. `implementation_status` records code reality. `verification_status` records QA coverage.

Implementation status values:

| Value | Meaning | Quality service behavior |
|---|---|---|
| implemented | Code exists and matches spec | Checks run normally |
| partial | Some behaviors implemented, others missing | Checks run against implemented_scope only |
| not_implemented | No code exists | Skipped; tracked as known gap |
| contradictory | Code exists but conflicts with spec | Flagged; not verified until resolved |

Partial implementation requires `implemented_scope` and `missing_scope`. Contradictory implementation requires a `conflict` block with `code_behavior`, `spec_behavior`, and `resolution`.

Verification status values:

| Value | Meaning |
|---|---|
| verified | Quality service actively checks this element |
| unverified | Element is verifiable but no QA check exists yet |
| not_observable | Element cannot be verified externally with current observation mechanisms |

The quality service runs checks only where `status: active` and `implementation_status: implemented | partial` and `verification_status: verified`.

## Verification Modes

Elements may be verifiable only under certain conditions:

| Mode | Meaning |
|---|---|
| runtime_monitor | Can be checked against a live production instance without mutation |
| generated_test | Requires test harness setup |
| both | Verifiable in both contexts |

Elements without `verification_mode` default to `runtime_monitor`.

## Severity

Every normative element carries a severity level: `critical`, `high`, `medium`, or `low`. Negative constraints default to `critical`; other elements default to `medium`.

## Source Traceability

Every element includes a `references` block linking it to the originating legacy spec section and implementing code:

```yaml
references:
  legacy_spec:
    file: telemetry.md
    section: "5. Chain Documentation"
  code:
    - file: dashboard/server.py
      line: 2397
```

During initial conversion, `legacy_spec` is required. Code references are populated during or after code-spec alignment verification.

## Document Sections

A spec document contains sections in any order. The compiler identifies sections by type markers, not position. Empty sections are omitted.

### 1. Overview

Non-normative context only.

### 2. Requirements

Normative. Every requirement is a single EARS-formatted statement with inspection metadata.

Required fields: `id`, `pattern`, `statement`, `severity`, `components`, `observe`, `check_type`, `verification_mode`, `verify`, `status`, `implementation_status`, `verification_status`, and `references`. Optional fields include `validator`, `assert`, `preconditions`, `time_constraint`, and `evidence`.

EARS patterns: `ubiquitous`, `event-driven`, `state-driven`, `unwanted`, `optional`.

### 3. Data Schemas

Normative JSON Schema definitions for all data structures.

### 4. Endpoint Contracts

Normative endpoint definitions with method, path, auth, request/response shapes, errors, and side effects.

### 5. Chain Event Schemas

Normative chain event type definitions with required fields and chain constraints.

### 6. State Machines

Normative lifecycle behaviors as state transition tables.

### 7. Negative Constraints

Normative absolute invariants: what the system must never do.

### 8. Workflow Contracts

Normative multi-step operations with correlation, ordered steps, timeout, and failure modes.

### 9. Schedule Contracts

Normative background behaviors with cadence, jitter, and failure handling.

### 10. Webhook Contracts

Normative externally triggered endpoints with signature verification and idempotency.

### 11. Derived Artifacts

Normative artifacts built from other artifacts with verifiable derivation.

### 12. Configuration Surface

Normative configuration that affects verifiable behavior.

### 13. Privacy Boundaries

Normative declarations of what data crosses what boundaries. The compiler auto-generates negative constraints from `sensitive_field_denylist` when `auto_generate_constraints: true`.

### 14. Non-Normative Sections

Clearly marked rationale, design history, future plans, and product philosophy. The spec compiler ignores these sections.

## Observation Metadata Reference

Every `observe` block uses structured metadata:

```yaml
observe:
  source: chain_record
  selector:
    event_type: telemetry_submitted
  correlation_key: artifact_hash
  role: dashboard
  observation_mode: direct
  capture_policy:
    body_hash_only: true
    retain_body: false
```

Source values:

| Value | How quality service accesses it |
|---|---|
| chain_record | Read chain file, filter by selector |
| api_response | Call endpoint, examine response |
| file_state | Read file, check content/existence/hash |
| config_state | Read config, check values |
| network_activity | Via observation_mode mechanism |
| ui_state | Browser automation or test_id inspection |
| process_state | Check process list, ports, resource usage |

For UI state, browser-level verification is needed for visibility, disabled states, and conditional rendering. For network activity, `observation_mode` is one of `app_event_log`, `egress_proxy_log`, or `packet_capture`.

Selector fields:

- `chain_record`: `event_type`, `event_type_pattern`, `policy_decision`, or any record field
- `api_response`: `endpoint`, `field`
- `file_state`: `base`, `path`, `field`
- `config_state`: `base`, `path`, `field`
- `network_activity`: `method`, `url`, `url_pattern`, `protocol`, `port`, `destination`
- `ui_state`: `route`, `component`, `test_id`, `viewport`
- `process_state`: `port`, `process_name`

Path base values:

| Base | Resolves to |
|---|---|
| runtime_root | The GOV_RUNTIME_DIR (`gov_runtime` by default) |
| repo_root | Repository root of governance-layer |
| website_root | Repository root of atested.com |
| configured | Explicitly configured path |

## Preconditions Reference

Generated-test checks declare setup requirements such as `runtime_fixture`, `clock: controllable`, `network_stub`, `dashboard_running`, `bearer_token`, and `network_available`.

## Expression Language

Machine-evaluable expressions use CEL:

```yaml
assert:
  - "payload.artifact_hash == sha256(observed_network_body)"
  - "record.prev_record_hash == previous_record.record_hash"
```

## Check Type Reference

| Value | Meaning |
|---|---|
| event_triggered | Matching observable artifact appears |
| periodic | Schedule-based |
| reconstructive | Replays inputs through reference evaluator |
| on_demand | Explicitly requested |
| transactional | Multi-step workflow completed or timed out |
| scheduled | Background schedule cadence check |
| cross_component | Cross dashboard/worker/proxy consistency |
| consistency | Derived artifact matches source |

## Spec Compiler Contract

The compiler produces verification rules, build-time tests, coverage manifests, schema validators, state machine monitors, workflow monitors, schedule monitors, and auto-generated privacy constraints.

The compiler rejects specs that have normative elements without IDs, missing observation metadata, undefined references, unreachable state machine states, invalid EARS statements, invalid imports, circular workflow dependencies, invalid CEL expressions, or component identifiers outside the defined set.

## QA Chain Record Format

Every quality service verification run produces a QA chain record with counts, failures, coverage breakdown, severity, observation metadata, evidence hash, and chain constraints. The QA chain is separate from the governance chain and uses the same cryptographic properties.
